# 2. 为什么要先找渲染底座

因为你现在的问题不是“教案数据没有”，而是：

```text
数据从后端出来以后，被哪些壳层、哪些 renderer、哪些旧模板、哪些 fallback 重新包装了？
```

如果不先找底座，后面会继续出现：

```text
自由备课一个 renderer
上传备课一个 renderer
R97B 一套 HTML
真实壳层一套 state
右栏自己再拼一套
小教上下文又接另一套
```

这就是之前越修越乱的原因。

---

# 3. 渲染层应该分四层

建议以后统一叫：

```text
Layer 0：App Shell
Layer 1：Prep Room Workspace Shell
Layer 2：Render Surface / Canvas
Layer 3：Lesson Content Renderer
```

## Layer 0：App Shell

包括：

```text
师维 logo
顶部全局导航
用户状态
搜索
通知
全局模式切换
```

它不应该知道教学设计细节。

---

## Layer 1：Prep Room Workspace Shell

包括：

```text
备课室标题
左侧单元树
右侧辅助栏
底部小教输入
工具栏
当前工作状态
```

它知道：

```text
当前课题
当前模式
当前 ViewModel state
当前右栏内容
当前小教上下文
```

但它不应该生成教学正文。

---

## Layer 2：Render Surface / Canvas

也就是你红框里的主区域。

它负责：

```text
加载哪一种工作内容
安排左中右区域
提供滚动容器
维持主舞台宽度
管理当前阅读位置
处理当前 episode / block 选中状态
```

它是“容器”，不是教学内容生成器。

---

## Layer 3：Lesson Content Renderer

它负责把：

```text
single_lesson_template
```

渲染成：

```text
本课依据
学情分析
教学目标
重难点
教学准备
教学过程
学习单与评价
后记
```

这一层是后续字段渲染、小教修改、导出预览的基础。

---

# 4. 当前必须先定一个原则

```text
后端只产出 ViewModel / single_lesson_template。
前端 renderer 只渲染，不推理教学内容。
```

也就是说，渲染层不能再干这些事：

```text
生成学情
补目标
拼教师话术
兜底教学环节
生成小教建议
判断课型
```

它只能根据 ViewModel 显示：

```text
已有内容
来源状态
折叠状态
待教师确认
source_gap
provisional candidate
```

# 5. 字段渲染先别接

你说“字段渲染可以等主壳的渲染定义定下来之后再考虑”，这个判断正确。

顺序应该是：

```text
先定渲染底座
再定 single_lesson_template 如何进入壳层
再定 block_id / field_path / render_slot
最后再接字段级渲染和小教修改
```

如果现在先做字段渲染，会导致：

```text
字段知道自己要显示
但不知道应该显示在哪个壳层、哪个块、哪个阅读模式
```

又会乱。
